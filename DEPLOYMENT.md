# 🚀 Deployment & Independent Verification Guide

This covers (1) choosing a Lightning network, (2) deploying to Vercel, and
(3) how a third party verifies the widget works on-chain.

---

## 1. Choose your Lightning backend

The app is **network-agnostic** — the only difference between testnet and mainnet
is *which LNbits instance you point at*. The code does not change.

### Option A — Testnet / Signet (no real money)

Best for a safe demo. Trade-off: a third-party tester must obtain test sats and a
testnet-capable wallet before they can pay, which adds friction.

How to get a testnet LNbits wallet:

- **Self-host LNbits** against a testnet/signet node (LND or Core Lightning). This
  is the most reliable but heaviest option. See <https://docs.lnbits.org>.
- **Mutinynet (signet) tooling** — a popular developer signet with faucets; pair it
  with a self-hosted or community LNbits instance.
- Whatever instance you use, confirm the network by creating an invoice: the
  BOLT11 prefix tells you — `lntb…` = testnet, `lntbs…` = signet, `lnbc…` = mainnet.
  The app surfaces this automatically as `network` in the invoice response.

Set `NETWORK_LABEL=testnet` (or `signet`) for the on-widget badge.

### Option B — Mainnet, tiny sats (recommended for *independent* verification)

Real Bitcoin, amounts of **1–21 sats** (a fraction of a US cent). This is the
strongest proof: **anyone, with any Lightning wallet** (Phoenix, Wallet of Satoshi,
Alby, Cash App, etc.) can pay and instantly see the result — no test-coin hunt.

- Use any LNbits instance backed by a mainnet node (e.g. `https://demo.lnbits.com`,
  or self-hosted).
- Keep presets small: `CREATOR_PRESET_AMOUNTS=21,100,500`.
- Set `NETWORK_LABEL=mainnet`.

> **Recommendation:** for a demo that "must be verifiable on-chain and testable by
> a third party independently", Option B removes the most friction. The amounts are
> trivial. If real money is not acceptable, use Option A and provide testers with a
> testnet wallet + faucet link.

## 2. Deploy to Vercel

### a. Push the repo to GitHub

```bash
git init
git add .
git commit -m "Lightning tipping widget"
git branch -M main
git remote add origin https://github.com/<you>/lightning-tipping-widget
git push -u origin main
```

### b. Import into Vercel

1. Go to <https://vercel.com/new> and import the repo.
2. Framework preset is auto-detected as **Next.js** — no changes needed.
3. Add the **Environment Variables** (Settings → Environment Variables):

   ```
   LNBITS_URL=https://your-lnbits-instance
   LNBITS_INVOICE_KEY=your_invoice_read_key
   CREATOR_NAME=Alice
   CREATOR_HANDLE=alice@getalby.com
   CREATOR_MESSAGE=Support my work with a Lightning tip ⚡
   CREATOR_PRESET_AMOUNTS=21,100,500
   NETWORK_LABEL=mainnet
   NEXT_PUBLIC_BASE_URL=https://your-app.vercel.app
   ```

4. Click **Deploy**. SSL is automatic.
5. After the first deploy, set `NEXT_PUBLIC_BASE_URL` to the real Vercel URL and
   redeploy so the embed snippet shows the correct domain.

### c. Deploy via CLI (alternative)

```bash
npm i -g vercel
vercel            # first deploy / link project
vercel --prod     # promote to production
```

## 3. ✅ Independent verification (give this to your tester)

A third party can confirm the system works end-to-end **without any account**:

1. Open the deployed site: `https://your-app.vercel.app`.
2. Scroll to **Live demo**. Pick an amount (e.g. 21 sats) and click **⚡ Tip**.
3. A **QR code** and a `lnbc…` invoice appear. Optionally copy the invoice and
   **decode it** at a BOLT11 decoder to confirm the amount and destination.
4. Pay it with any Lightning wallet (scan the QR, or paste the invoice).
5. Within ~1 second the widget flips to **"Tip received!"**.
6. The tip appears in the **live tip feed** with the name/message entered.
7. Refresh the page — the tip is still there, because it is read back from the
   Lightning node's settled payment history (not from browser state).

What this proves:
- A **real Lightning invoice** was generated by a real node.
- A **real payment** settled on the Lightning Network.
- The UI reflects on-chain truth in **sub-second** time and persists it.

## 4. Production checklist

- [ ] `LNBITS_URL` + `LNBITS_INVOICE_KEY` set in Vercel (invoice key only)
- [ ] `NEXT_PUBLIC_BASE_URL` matches the deployed domain
- [ ] `NETWORK_LABEL` matches the actual node network
- [ ] Presets sized appropriately for the network
- [ ] Test the verification flow above on the live URL
- [ ] Embed snippet tested on a second site / blog

## 5. Troubleshooting

| Symptom | Likely cause / fix |
| --- | --- |
| "Could not create invoice" | Wrong `LNBITS_URL` or `LNBITS_INVOICE_KEY`; instance down |
| Widget loads but feed is empty | Normal until the first tip settles |
| Embed shows nothing on a site | Ensure the page allows iframes; check the `<script>` src URL |
| Invoice never flips to paid | Payment didn't settle; check the wallet and the LNbits wallet history |
| Wrong network badge | Set `NETWORK_LABEL`; the real network is whatever your node runs |
